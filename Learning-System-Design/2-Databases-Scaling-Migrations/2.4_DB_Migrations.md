# Database Migrations

## Definition
Database migration is the process of **changing the database schema or data structure over time** in a controlled and versioned manner.

---

## Why Migrations are Needed

- Add/remove columns or tables
- Modify schema without downtime
- Evolve system with new features
- Maintain consistency across environments

---

## Types of Migrations

### 1. Schema Migration
Changes structure of database

Examples:
- Add column
- Drop table
- Modify data type

---

### 2. Data Migration
Moves or transforms existing data

Examples:
- Backfilling new column
- Format changes
- Data cleanup

---

## Migration Strategies

### 1. Offline Migration

- Stop application
- Apply migration
- Restart system

Pros:
- Simple

Cons:
- Downtime required

---

### 2. Online Migration

- System remains live during migration

Techniques:
- Backward-compatible changes
- Gradual rollout

---

## Backward Compatible Changes

Safe changes that don’t break existing code:

- Add new column (nullable or default)
- Add new table
- Add index

---

## Breaking Changes (Avoid Directly)

- Dropping columns
- Renaming columns
- Changing data types

Require multi-step migration

---

## Expand and Contract Pattern

### Step 1: Expand
- Add new schema (e.g., new column)
- Keep old schema intact

### Step 2: Migrate Data
- Backfill data to new structure

### Step 3: Switch
- Update application to use new schema

### Step 4: Contract
- Remove old schema

---

## Example

Renaming column `name` → `full_name`

1. Add `full_name`
2. Copy data from `name` → `full_name`
3. Update code to use `full_name`
4. Remove `name`

---

## Versioning Migrations

Each migration has a version:

- V1 → Create table
- V2 → Add column
- V3 → Add index

Maintained using migration tools.

---

## Idempotency

Migration should be safe to run multiple times.

Example:
- Check if column exists before adding

---

## Rollbacks

Ability to revert migration if failure occurs.

Example:
- Drop added column
- Restore previous schema

---

## Handling Large Data

- Use batch processing
- Avoid full table locks
- Use background jobs

---

## Zero Downtime Migration Techniques

- Dual writes (write to old + new schema)
- Feature flags
- Read from both schemas during transition

---

## Index Migrations

- Create indexes in background
- Avoid blocking queries

---

## Common Tools

- Flyway
- Liquibase
- Alembic
- Django Migrations

---

## Challenges

- Data consistency
- Downtime risk
- Long-running migrations
- Rollback complexity

---

## Best Practices

- Keep migrations small and incremental
- Test on staging before production
- Use backward-compatible changes
- Monitor during deployment
- Always have rollback plan

---

## Key Takeaways

- Migrations evolve database schema safely
- Prefer online, backward-compatible changes
- Use expand-contract pattern for breaking changes
- Ensure versioning, idempotency, and rollback support
