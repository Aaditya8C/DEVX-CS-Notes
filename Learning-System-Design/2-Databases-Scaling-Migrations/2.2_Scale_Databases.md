# Scaling Databases – Core Techniques

This document covers key techniques used to scale databases efficiently under high read/write load.

---

# 1. Request Condensing

## Definition
Request condensing combines **multiple identical requests into a single request** to reduce redundant work on the database.

---

## How It Works
- Multiple clients request the same data
- System merges them into one request
- Result is shared among all waiting clients

---

## Example
- 100 users request the same product details simultaneously
- Instead of 100 DB queries → only 1 query is executed
- Response is reused for all

---

## Use Cases
- Cache stampede prevention
- High read systems (e.g., trending posts)

---

## Time Complexity

| Operation | Complexity |
|----------|-----------|
| Request handling | O(1) (dedup lookup) |
| DB query | Reduced from O(N) → O(1) |

---

## Benefit
- Reduces database load
- Improves latency under high concurrency

---

# 2. Log using Linked List (Write-Ahead Log Style)

## Definition
Writes are appended sequentially like a **linked list (log)** instead of updating data in-place.

---

## How It Works
- Each write is appended to the end
- Data is not overwritten
- Reads may require scanning or indexing

---

## Example
- Insert operations in log-based storage
- Event sourcing systems

---

## Use Cases
- Write-heavy systems
- Logging systems
- Append-only databases

---

## Time Complexity

| Operation | Complexity |
|----------|-----------|
| Write | O(1) |
| Read (naive) | O(N) |

---

## Benefit
- Very fast writes
- Sequential disk access (efficient for SSD/HDD)

---

## Drawback
- Slow reads without indexing
- Requires compaction later

---

# 3. B+ Tree vs Linked List

## B+ Tree

### Definition
A balanced tree used in databases for indexing and storage.

### Properties
- Sorted data
- All data stored in leaf nodes
- Leaf nodes are linked

---

### Example
- MySQL InnoDB index
- PostgreSQL index

---

### Time Complexity

| Operation | Complexity |
|----------|-----------|
| Search | O(log N) |
| Insert | O(log N) |
| Range Query | O(log N + K) |

---

### Benefit
- Efficient range queries
- Good read performance

---

## Linked List (Log)

### Definition
Sequential append-only structure

---

### Time Complexity

| Operation | Complexity |
|----------|-----------|
| Insert | O(1) |
| Search | O(N) |

---

## Comparison

| Feature | B+ Tree | Linked List |
|--------|--------|------------|
| Writes | Slower (log N) | Fast (O(1)) |
| Reads | Fast | Slow |
| Range Queries | Efficient | Inefficient |
| Storage | Structured | Append-only |

---

# 4. Sorted String Table (SSTable)

## Definition
An SSTable is an **immutable, sorted file of key-value pairs** stored on disk.

---

## Properties
- Data is sorted by key
- Once written → never modified
- Stored as files on disk

---

## How It Works
- Writes go to in-memory structure (memtable)
- Flushed to disk as SSTable
- Multiple SSTables exist

---

## Example
- Cassandra
- LevelDB
- RocksDB

---

## Time Complexity

| Operation | Complexity |
|----------|-----------|
| Write (memtable) | O(1) |
| Flush to disk | O(N log N) |
| Read | O(log N) (binary search) |

---

## Benefit
- Efficient sequential writes
- Good for large-scale systems

---

## Drawback
- Multiple files → read amplification
- Needs compaction

---

# 5. Compaction

## Definition
Compaction is the process of **merging multiple SSTables into fewer optimized ones**.

---

## Why Needed
- Remove duplicate keys
- Delete obsolete data
- Improve read performance

---

## Types

### 1. Minor Compaction
- Merge small SSTables

### 2. Major Compaction
- Merge all SSTables into one

---

## Example
- Cassandra compaction
- LSM tree systems

---

## Time Complexity

| Operation | Complexity |
|----------|-----------|
| Compaction | O(N) |
| Read (after compaction) | Improved |

---

## Benefit
- Reduces read amplification
- Frees storage
- Improves query speed

---

# How These Fit Together (Big Picture)

Modern databases (LSM Tree based):

1. Writes → appended (log / memtable)
2. Data → flushed as SSTables
3. Multiple SSTables → merged via compaction
4. Bloom filters → used for fast existence check

---

# Key Takeaways

- Request condensing reduces duplicate DB queries
- Linked list (log) enables fast writes
- B+ trees optimize read-heavy workloads
- SSTables enable scalable storage
- Compaction maintains system efficiency
