# Thrashing

## Definition

Thrashing is a condition in a virtual memory system where **CPU utilization drops drastically because the system spends most of its time handling page faults instead of executing processes**.

It occurs when the operating system continuously swaps pages between **RAM and disk** due to insufficient memory frames.

---

## How Thrashing Happens

1. The system attempts to increase CPU utilization by **adding more processes**.
2. Each process requires **memory frames**.
3. The total demand for frames exceeds available physical memory.
4. Processes begin to experience **frequent page faults**.
5. The OS continuously performs **paging operations (disk I/O)**.
6. CPU waits for disk operations instead of executing instructions.

**Result:** CPU utilization drops sharply.

---

## Symptoms of Thrashing

- Very **high page fault rate**
- **Low CPU utilization**
- Heavy **disk I/O activity**
- Large **paging queues**
- Significant **performance degradation**

---

# Causes of Thrashing

### 1. High Degree of Multiprogramming

### 2. Insufficient Memory Frames

### 3. Frequent Page Faults

### 4. Global Page Replacement

---

# Working Set Model

The **Working Set Model** is used to prevent thrashing by ensuring each process has enough frames to hold its actively used pages.

It is based on the **principle of locality of reference**.

---

## Working Set Definition

A **Working Set** is the set of pages referenced by a process during a fixed time window **Δ (delta)**.

If a page has been referenced within the last **Δ time units**, it belongs to the working set.

---

## Working Set Size
WSS_i = Number of pages used by process i during time window Δ


Where:

- `WSS_i` = Working set size of process *i*

---

## Total Memory Demand
D = Σ WSS_i


Where:

- `D` = total number of frames required by all processes
- `WSS_i` = working set size of process *i*

---

## Thrashing Condition
If D > Available Frames → Thrashing occurs


When the total memory demand exceeds available frames, processes continuously replace each other's pages.

---

# Techniques to Prevent Thrashing

## 1. Local Replacement

Each process can only replace **its own pages** instead of pages belonging to other processes.

**Advantage**

- Prevents **frame stealing**
- Reduces cascading page faults

---

## 2. Working Set Strategy

The OS ensures each process receives **at least its working set size in frames**.

If memory is insufficient:

- Some processes are **temporarily suspended**
- Frames are freed
- Remaining processes execute efficiently without thrashing

---

## Key Takeaways

- Thrashing occurs when the system spends **more time paging than executing processes**.
- Main cause: **insufficient memory frames for active processes**.
- Identified by **high page fault rate and low CPU utilization**.
- Prevented using:
  - **Working Set Model**
  - **Local Page Replacement**
  - **Reducing degree of multiprogramming**
