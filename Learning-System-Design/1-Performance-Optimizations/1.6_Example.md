# Case Study: Load Balancing in Netflix

## Overview

Netflix is a global streaming platform serving **millions of users simultaneously** across different regions. To ensure **high availability, low latency, and scalability**, Netflix uses a **multi-layered load balancing architecture**.

---

## Problem Statement

Netflix faces the following challenges:

- Massive traffic spikes (e.g., new show releases)
- Global user base with varying latency
- Need for high availability (no downtime)
- Handling millions of requests per second

Without load balancing:

- Servers would get overloaded
- High latency and buffering issues
- System failures during peak traffic

---

## High-Level Architecture

Client → DNS → Load Balancer → API Gateway → Microservices → Database/CDN

---

## Load Balancing Strategy

Netflix uses **multi-level load balancing**:

---

### 1. Global Load Balancing (DNS Level)

- Uses DNS to route users to the **nearest geographic region**
- Initially used geo-based routing, later improved to **latency-based routing** :contentReference[oaicite:0]{index=0}

Purpose:
- Reduce latency
- Improve user experience

---

### 2. Edge Load Balancing (CDN Level)

- Netflix uses its own CDN called **Open Connect**
- Users are routed to the nearest CDN server

Benefits:
- Faster content delivery
- Reduced network congestion
- High-quality video streaming :contentReference[oaicite:1]{index=1}

---

### 3. Application Load Balancing (AWS ELB)

- Uses **AWS Elastic Load Balancer (ELB)** to distribute traffic across services
- Load is balanced:
  - Across availability zones
  - Across instances inside zones :contentReference[oaicite:2]{index=2}

Features:
- Health checks
- Automatic failover
- Even traffic distribution

---

### 4. Microservices-Level Load Balancing

- Netflix uses **client-side load balancing** (e.g., Ribbon)
- Requests are distributed across service instances

Benefits:
- Reduces dependency on central load balancer
- Improves scalability in microservices architecture :contentReference[oaicite:3]{index=3}

---

### 5. API Gateway Load Balancing

- Uses **Zuul (API Gateway)** for routing
- Handles:
  - Request routing
  - Security
  - Traffic filtering

Scale:
- Handles **millions of requests per second** :contentReference[oaicite:4]{index=4}

---

## Key Techniques Used

- Round-robin and weighted routing
- Health checks and failover
- Auto-scaling with cloud infrastructure
- Latency-based routing
- Multi-layer load balancing

---

## Benefits Achieved

- Handles **massive traffic spikes efficiently**
- Provides **low latency streaming globally**
- Ensures **high availability and fault tolerance**
- Scales dynamically based on demand :contentReference[oaicite:5]{index=5}

---

## Key Learnings

- Use **multi-layer load balancing** (DNS + LB + microservices)
- Combine **global + local routing strategies**
- Integrate load balancing with **auto-scaling**
- Prefer **client-side load balancing** in microservices
- Always include **health checks and failover mechanisms**

---

## Summary

Netflix uses a **layered load balancing approach** combining:

- DNS-based global routing
- CDN-level distribution
- Cloud load balancers (ELB)
- Client-side load balancing in microservices

This architecture enables Netflix to deliver a **highly scalable, reliable, and low-latency streaming experience worldwide**.
