# Case Study: E-Commerce Order System (Consistency + Isolation)

## Overview

Consider an e-commerce platform (like Amazon/Flipkart) handling:

- User orders
- Payments
- Inventory updates
- Order history

This system must balance:
- Consistency
- Availability
- Performance

---

# System Components

- Order Service
- Payment Service
- Inventory Service
- Database (distributed + replicated)

---

# Scenario: Placing an Order

User buys a product:

Steps:
1. Check inventory
2. Reserve item
3. Process payment
4. Confirm order

---

# Consistency in This System

## Problem

Data is distributed across services:
- Inventory DB
- Orders DB
- Payments DB

---

## Example Issue

User places order:

- Inventory updated (stock reduced)
- Payment fails

❗ System inconsistency:
- Item reserved but order failed

---

## Solution

Use:
- Strong consistency (for payments, inventory)
- Eventual consistency (for analytics, recommendations)

---

# Consistency Levels Used

## Strong Consistency (Critical Path)

Used in:
- Payment processing
- Inventory deduction

Example:
- When item stock = 1
- Two users try to buy

System ensures:
- Only one succeeds

---

## Eventual Consistency (Non-Critical)

Used in:
- Order history
- Notifications
- Recommendations

Example:
- Order placed → appears in history after few seconds

---

## Causal Consistency

Used in:
- Order tracking

Example:
- "Order placed" → "Order shipped"

User must see:
- Correct order of events

---

# Transaction Isolation in This System

---

## Problem: Concurrent Orders

Stock = 1

Two users:
- User A
- User B

---

## Without Isolation

Both read:
- Stock = 1

Both place order

❗ Result:
- Overselling (stock becomes -1)

---

## With Isolation Levels

---

### Read Committed (Common)

- Users only see committed data

Still possible:
- Two users read stock = 1 before update

❗ Overselling possible

---

### Repeatable Read

- Same transaction sees same data

Still possible:
- Phantom issue (new orders)

---

### Serializable (Used for Critical Section)

System enforces:

- Only one transaction executes at a time

Flow:
- User A locks row
- User B waits

Result:
- Only one order succeeds

---

# Anomalies in Real System

---

## Dirty Read (Avoided)

Payment not completed but order read as success

---

## Non-Repeatable Read

User refreshes cart:
- Sees different prices/stock

---

## Phantom Read

User sees:
- 1 item available

Another user buys → new state appears

---

# Practical Design Decisions

---

## Where to Use Strong Consistency

- Payments
- Inventory updates

Reason:
- Prevent money loss / overselling

---

## Where to Use Eventual Consistency

- Order history
- Notifications
- Analytics

Reason:
- Better performance
- Not critical if slightly delayed

---

## Hybrid Approach

Real systems use mix:

| Component        | Consistency Level |
|----------------|------------------|
| Payment        | Strong           |
| Inventory      | Strong           |
| Order History  | Eventual         |
| Notifications  | Eventual         |

---

# Techniques Used

- Distributed transactions (or saga pattern)
- Row-level locking
- Idempotency (retry safety)
- Message queues for async updates

---

# Key Takeaways

- Real systems use **different consistency levels for different components**
- Strong consistency is used only where correctness is critical
- Isolation levels prevent concurrency issues like overselling
- Serializable isolation is used for critical sections
- Eventual consistency improves scalability and performance
