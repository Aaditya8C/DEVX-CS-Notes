# Transaction Isolation Levels

## Definition
Transaction isolation defines **how transactions interact with each other when executed concurrently**.

It controls **visibility of changes made by one transaction to others**.

---

## Why Isolation is Needed

Without isolation, concurrent transactions can lead to:
- Incorrect data
- Inconsistent reads
- Race conditions

---
# Common Anomalies (with Examples)

## 1. Dirty Read

### Definition
Reading **uncommitted data** from another transaction.

---

### Example

Initial balance = 1000

T1:
- Update balance → 500 (not committed)

T2:
- Reads balance → 500 (dirty read)

T1:
- Rollback → balance back to 1000

❗ T2 read invalid data

---

## 2. Non-Repeatable Read

### Definition
Reading the same row twice gives **different values** within a transaction.

---

### Example

Initial balance = 1000

T1:
- Reads balance → 1000

T2:
- Updates balance → 500 (commit)

T1:
- Reads again → 500

❗ Same query, different result

---

## 3. Phantom Read

### Definition
Repeated query returns **different number of rows**.

---

### Example

Initial:
Accounts with balance > 1000 → 2 rows

T1:
- SELECT * WHERE balance > 1000 → 2 rows

T2:
- INSERT new account with balance = 2000 (commit)

T1:
- SELECT again → 3 rows

❗ New row "appears" (phantom)

---

# Isolation Levels (Low → High)

---

## 1. Read Uncommitted

### Definition
Transactions can read data **even if it is not committed**.

### Allowed Anomalies
- Dirty Read ✅
- Non-repeatable Read ✅
- Phantom Read ✅

---

### Example

Transaction T1:
- Updates balance = 1000 → 500 (not committed)

Transaction T2:
- Reads balance = 500

If T1 rolls back → T2 read invalid data

---

### Use Case
- Rarely used
- Systems where speed matters more than accuracy

---

## 2. Read Committed

### Definition
Transactions can only read **committed data**.

### Allowed Anomalies
- Dirty Read ❌
- Non-repeatable Read ✅
- Phantom Read ✅

---

### Example

T1:
- Updates balance = 1000 → 500 (commits)

T2:
- Reads before commit → sees 1000
- Reads after commit → sees 500

Same query → different result

---

### Use Case
- Default in many databases (e.g., PostgreSQL)

---

## 3. Repeatable Read

### Definition
If a transaction reads a row, it will **always see the same value** during its execution.

### Allowed Anomalies
- Dirty Read ❌
- Non-repeatable Read ❌
- Phantom Read ✅ (in some systems)

---

### Example

T1:
- Reads balance = 1000

T2:
- Updates balance to 500 (commits)

T1:
- Reads again → still sees 1000

---

### Use Case
- Systems requiring stable reads

---

## 4. Serializable

### Definition
Transactions behave as if they are executed **one after another (no concurrency issues)**.

### Allowed Anomalies
- Dirty Read ❌
- Non-repeatable Read ❌
- Phantom Read ❌

---

### Example

T1 and T2 execute:

Instead of interleaving:
- T1 fully executes
- Then T2 executes

---

### Use Case
- Banking systems
- Critical financial operations

---

## Comparison Table

| Isolation Level     | Dirty Read | Non-Repeatable Read | Phantom Read |
|--------------------|-----------|---------------------|--------------|
| Read Uncommitted   | Yes       | Yes                 | Yes          |
| Read Committed     | No        | Yes                 | Yes          |
| Repeatable Read    | No        | No                  | Yes*         |
| Serializable       | No        | No                  | No           |

(*depends on implementation)

---

## Trade-offs

| Level              | Consistency | Performance |
|-------------------|------------|-------------|
| Read Uncommitted  | Low        | High        |
| Read Committed    | Medium     | High        |
| Repeatable Read   | High       | Medium      |
| Serializable      | Very High  | Low         |

---

## Key Takeaways

- Isolation controls **how transactions see each other’s data**
- Higher isolation → better consistency but lower performance
- Serializable is safest but slowest
- Most systems use **Read Committed or Repeatable Read**
