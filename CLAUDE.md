<!-- # Repository Instructions

This repository contains my personal study notes for Computer Science and Software Engineering.

## Goal

Generate concise, interview-oriented Markdown notes that are easy to revise.

The notes should focus on **understanding**, **interview preparation**, and **quick revision**, rather than long theoretical explanations.

---

# Markdown Style

- Use Markdown only.
- Use clear section headings.
- Keep explanations concise.
- Avoid unnecessary paragraphs.
- Use horizontal separators (---) between major topics.
- Prefer bullet points over large paragraphs.
- Use tables whenever comparisons are helpful.
- Use code blocks only for:
  - formulas
  - architecture flows
  - pseudocode
- Do not add emojis.

---

# Explanation Style

For every concept, explain in this order whenever applicable:

1. Definition
2. Why it is needed
3. How it works
4. Example (real-world or practical)
5. Time Complexity (if applicable)
6. Use Cases

Skip sections that are not applicable.

---

# Examples

Prefer practical software engineering examples instead of generic examples.

Examples include:

- Amazon
- Netflix
- Uber
- WhatsApp
- Google Maps
- Banking Systems
- Distributed Databases
- E-commerce
- Payment Systems

Avoid overly simplistic examples unless they help explain the concept.

---

# Revision Style

The notes are intended for revision.

Therefore:

- Keep explanations precise.
- Avoid repeating the same information.
- Avoid unnecessary theory.
- Highlight only interview-relevant points.

---

# File Organization

- Group related concepts into a single Markdown file whenever reasonable.
- Do not create many tiny files.
- Create separate files only for large independent topics.

---

# Diagrams

Whenever a flow or architecture improves understanding, use simple text diagrams or mermaid diagrams.

Example:

Client
    │
Load Balancer
    │
Application Servers
    │
Database

Do not create ASCII art beyond simple flows.

---

# Tables

Use comparison tables whenever useful.

Examples:

- SQL vs NoSQL
- Strong vs Eventual Consistency
- B+ Tree vs LSM Tree
- Kafka vs RabbitMQ

---

# Mathematics

Whenever formulas are involved:

- Write the formula.
- Explain every variable.
- Explain what the formula represents.
- Mention why it is important.

---

# Real-World Systems

Whenever possible, include one practical case study showing where the concept is used in production systems.

Keep the case study concise and relevant.

---

# What to Avoid

- Long textbook explanations
- Excessive history
- Repeated definitions
- Unnecessary markdown styling
- Overly verbose examples
- Creating many small files

---

# Objective

The final output should resemble high-quality interview revision notes that can be read in 3–5 minutes per topic while still providing sufficient depth to understand the concept. -->
