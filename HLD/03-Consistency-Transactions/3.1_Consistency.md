# Consistency in Distributed Systems

## What is Consistency?

Consistency ensures that **all nodes see the same data at the same time**.

After a write:
- Strong consistency → all reads return latest value
- Weak consistency → reads may return stale data

---

## The Simplest Case (Single Node)

Single database:

Client → Database

- One source of truth
- No replication issues
- Strong consistency guaranteed

---

## Single Node Problems

- Limited scalability
- Single point of failure
- Cannot handle large traffic
- No fault tolerance

---

## Splitting the Data (Sharding)

Data is partitioned across multiple nodes.

Example:
- Users A–M → Node 1
- Users N–Z → Node 2

Benefits:
- Horizontal scaling
- Reduced load per node

---

## Problems with Disjoint Data

When data is split:

- Cross-shard queries become complex
- Joins across shards are expensive
- Transactions across shards are hard

Example:
- Transfer money between users on different shards

---

## Data Copies (Replication)

Data is duplicated across multiple nodes.

Example:
- Primary → Replica 1 → Replica 2

Benefits:
- Fault tolerance
- Improved read performance

---

## Consistency Issue in Replication

- Writes go to primary
- Replicas may lag

Problem:
- Reads from replica may return stale data

---

## The Two Generals Problem

### Idea
Two parties must coordinate over unreliable communication.

Problem:
- Messages may be lost
- No guarantee both sides agree

Implication:
- Perfect agreement (consensus) is impossible in unreliable systems

---

## Leader Assignment

One node is chosen as **leader (primary)**.

- All writes go to leader
- Followers replicate data

Benefits:
- Simplifies consistency
- Avoids write conflicts

Problems:
- Leader failure handling required
- Failover complexity

---

## Consistency Tradeoffs

From CAP Theorem:

You can choose only two:

- Consistency (C)
- Availability (A)
- Partition Tolerance (P)

---

### Types of Systems

- CP (Consistency + Partition Tolerance)
  - May reject requests during failures

- AP (Availability + Partition Tolerance)
  - Allows stale reads

---

## Two-Phase Commit (2PC)

### Purpose
Ensure atomic transactions across multiple nodes.

---

### Phase 1: Prepare
- Coordinator asks all nodes: "Can you commit?"
- Nodes respond YES/NO

---

### Phase 2: Commit / Abort
- If all YES → commit
- If any NO → abort

---

### Properties

- Guarantees atomicity
- Ensures consistency across nodes

---

### Drawbacks

- Blocking protocol
- Coordinator failure can stall system
- Slow (multiple network round trips)

---

## Eventual Consistency

### Definition
System guarantees that **all replicas will eventually converge to the same value**, given no new updates.

---

### Characteristics

- Reads may return stale data
- System remains highly available
- Data becomes consistent over time

---

### Example

- Write happens at time T
- Some replicas update later
- Eventually all replicas sync

---

### Use Cases

- Social media feeds
- DNS systems
- Distributed caches

---

## Strong vs Eventual Consistency

| Feature | Strong Consistency | Eventual Consistency |
|--------|------------------|---------------------|
| Read freshness | Always latest | May be stale |
| Latency | Higher | Lower |
| Availability | Lower | Higher |
| Complexity | High | Moderate |

---

## Key Takeaways

- Consistency ensures all nodes see the same data
- Single node → simple but not scalable
- Replication introduces consistency challenges
- Leader-based systems simplify writes
- 2PC ensures atomic transactions but is slow
- Eventual consistency trades accuracy for availability
- Distributed systems must balance consistency, availability, and partition tolerance
