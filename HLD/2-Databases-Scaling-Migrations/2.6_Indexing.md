# Database Indexing

---

## What is an Index?

A separate data structure the DB maintains to make lookups **faster** — at the cost of extra storage and slower writes.

```
  Without index: SELECT * WHERE email = 'x@g.com'
  → Scans all 10M rows one by one — O(N) — seconds

  With index on email:
  → Jumps directly to matching rows — O(log N) — milliseconds
```

---

## Types of Indexes

### 1. B-Tree Index (Default)
Balanced tree. Good for **equality and range** queries.
- Used by PostgreSQL, MySQL for most indexes
- `WHERE age > 20` — efficient
- `ORDER BY created_at` — efficient

### 2. Hash Index
Hash function maps value → row location. O(1) for exact match, **no range queries**.
- `WHERE email = 'x@g.com'` — fastest
- `WHERE email > 'x@g.com'` — not supported

### 3. Composite Index (Multi-Column)
Index on two or more columns together.

```
  CREATE INDEX idx_city_age ON users(city, age);

  Efficient:   WHERE city = 'Pune'                    (leftmost column)
  Efficient:   WHERE city = 'Pune' AND age > 20
  Inefficient: WHERE age > 20                         (skips leftmost — index unused)
```

> **Left-prefix rule**: An index on (A, B, C) only helps queries starting with A.

### 4. Full-Text Index (Inverted Index)
For searching inside text content. Powers search engines.

```
  Doc 1: "System design is fun"
  Doc 2: "System design at scale"

  Index: "system" → [Doc1, Doc2],  "fun" → [Doc1]
  Query: SEARCH "system" → [Doc1, Doc2]
```
Used by: Elasticsearch, PostgreSQL FTS.

### 5. Partial Index
Index only a subset of rows matching a condition — saves space.

```sql
  CREATE INDEX idx_active ON users(email) WHERE is_active = TRUE;
  -- Indexes only active users, not all 100M rows
```

---

## Performance Impact

| Operation | With Index | Without Index |
|---|---|---|
| Read (WHERE lookup) | Fast — O(log N) | Slow — O(N) full scan |
| Write (INSERT/UPDATE/DELETE) | Slower — updates all indexes too | Faster |
| Storage | More | Less |

> Too many indexes on a write-heavy table = write bottleneck.

---

## When to Add an Index

Add index on columns that are:
- Frequently used in `WHERE` clauses
- Used in `JOIN` conditions (foreign keys)
- Used in `ORDER BY` or `GROUP BY`
- High cardinality (many unique values — e.g., email, phone)

Do NOT index:
- Low cardinality columns (e.g., gender — only 2 values, index gives no benefit)
- Rarely queried columns
- Write-heavy tables with many existing indexes

---

## Indexes in Sharded Databases

Each shard has its own **local index**. Queries without the shard key must scatter to all shards.

```
  Sharded by user_id:
  Query: WHERE email = 'x@g.com'   → must query ALL shards (slow)
  Query: WHERE user_id = 123       → hits exactly 1 shard (fast)
```

> Design queries to always include the shard key.

---

## Key Takeaways

- B-Tree is the default — good for equality and range
- Hash index is O(1) for exact match only — no range support
- Composite index follows the left-prefix rule
- Full-text (inverted) index powers search engines
- Use `EXPLAIN` to verify the DB is actually using your index
- Every extra index slows down writes — add only what you need
