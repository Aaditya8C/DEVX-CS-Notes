# Database Indexing

---

## What is an Index?

An **index** is a separate data structure that the database maintains alongside your table to make lookups **faster** — at the cost of extra storage and slower writes.

Without an index, a query scans **every row** in the table (full table scan) — O(N).
With an index, the database can find rows in O(log N) or O(1).

```
  Table: 1 million users
  Query: SELECT * FROM users WHERE email = 'a@gmail.com'

  WITHOUT index:
  → Scans all 1,000,000 rows one by one
  → Takes: seconds

  WITH index on email:
  → Jumps directly to the row via index
  → Takes: milliseconds
```

---

## How an Index Works Internally (B-Tree)

Most database indexes use a **B-Tree (Balanced Tree)** structure.

```
  Index on email column:

              [m@gmail.com]
              /             \
    [c@gmail.com]        [s@gmail.com]
     /        \              /       \
  [a@]       [e@]        [r@]       [z@]
   ↓           ↓           ↓          ↓
  Row 4       Row 9      Row 2      Row 7   ← Pointers to actual rows
```

- Tree stays **balanced** — lookups are always O(log N)
- Range queries (`WHERE email > 'a' AND email < 'm'`) are efficient — traverse in order
- Most popular index type in: **PostgreSQL, MySQL, Oracle**

## Types of Indexes

### 1. Primary Index (Clustered Index)

- Built on the **Primary Key** automatically
- The actual data rows are **physically ordered** by this key
- Only **one per table** (rows can only be in one physical order)

```
  Table rows physically stored in order of id:
  [id=1, name=A] [id=2, name=B] [id=3, name=C] ...
                                      ↑
                          Clustered — data IS the index
```

**Benefit:** Range queries on primary key are extremely fast.

---

### 2. Secondary Index (Non-Clustered Index)

- Built on **any other column** (email, phone, name, etc.)
- Index stores the column value + **pointer to the actual row**
- Can have **many per table**

```
  Table: users (id, name, email, city)

  Secondary index on email:
  ┌────────────────────┬──────────┐
  │ email (sorted)     │ row ptr  │
  ├────────────────────┼──────────┤
  │ a@gmail.com        │ → row 4  │
  │ b@gmail.com        │ → row 12 │
  │ c@gmail.com        │ → row 7  │
  └────────────────────┴──────────┘

  Lookup: WHERE email = 'b@gmail.com'
  → Find in index → get row pointer → fetch actual row
```

---

### 3. Composite Index (Multi-Column Index)

- Index on **two or more columns together**
- Useful when queries filter on multiple columns

```
  CREATE INDEX idx_city_age ON users(city, age);

  Efficient queries:
  WHERE city = 'Pune'                    Yes (leftmost column)
  WHERE city = 'Pune' AND age > 20       Yes (both columns)
  WHERE city = 'Pune' AND age = 21       Yes

  Inefficient queries:
  WHERE age = 21                         No (skips leftmost column)
  WHERE age > 20                         No (leftmost column not used)
```

> **Left-prefix rule**: A composite index on (A, B, C) only helps queries that filter on A, or A+B, or A+B+C. Skipping A makes the index useless.

---

### 4. Hash Index

- Uses a **hash function** to map column value → row location
- O(1) lookups for **exact match** queries

```
  hash("a@gmail.com") → bucket 42 → row location

  Query: WHERE email = 'a@gmail.com'  Yes Extremely fast
  Query: WHERE email > 'a@gmail.com'  No Hash doesn't support ranges
```

| Property | B-Tree Index | Hash Index |
|---|---|---|
| Equality queries | Yes Fast (O log N) | Yes Fastest (O(1)) |
| Range queries | Yes Efficient | No Not supported |
| Sorting | Yes Yes | No No |
| Used in | PostgreSQL, MySQL | Redis, Memcached |

---

### 5. Full-Text Index (Inverted Index)

- Used for **searching text content** — not exact matches
- Used by Elasticsearch, PostgreSQL's full-text search

```
  Documents:
  Doc 1: "System design is fun"
  Doc 2: "System design at scale"
  Doc 3: "Fun with databases"

  Inverted Index:
  "system"  → [Doc 1, Doc 2]
  "design"  → [Doc 1, Doc 2]
  "fun"     → [Doc 1, Doc 3]
  "scale"   → [Doc 2]

  Query: SEARCH "system design"
  → Lookup "system" → [1,2], "design" → [1,2] → intersection → [1,2]
```

**Used by:** Elasticsearch, Solr, PostgreSQL FTS, MySQL FULLTEXT

---

### 6. Partial Index

- Index only a **subset** of rows that match a condition
- Saves space and is faster to maintain

```sql
  CREATE INDEX idx_active_users ON users(email) WHERE is_active = TRUE;

  -- Only indexes active users, not all 100M users
  -- Query: WHERE email = 'x@g.com' AND is_active = TRUE  → uses partial index
```

## How Indexes Affect Performance

### Read Performance (Faster Yes)

```
  Without index: Full table scan O(N) — reads every row
  With index:    B-Tree lookup O(log N) or Hash O(1)

  For 10M rows:
  Full scan:   ~10 seconds
  B-Tree scan: ~milliseconds
```

### Write Performance (Slower No)

Every `INSERT`, `UPDATE`, `DELETE` must also **update all indexes** on that table.

```
  Table has 5 indexes on it:
  INSERT one row → update table + update 5 indexes
  → 6 write operations instead of 1

  Too many indexes on a write-heavy table = performance bottleneck
```

### Storage (More space No)

Indexes are stored separately and can be as large as the table itself.

## Index Best Practices

### When to Add an Index

```
  Yes Columns used in WHERE clauses frequently
  Yes Columns used in JOIN conditions (foreign keys)
  Yes Columns used in ORDER BY or GROUP BY
  Yes Columns with high cardinality (many unique values)

  Example: email, phone, user_id — high cardinality → index helps a lot
```

### When NOT to Add an Index

```
  No Write-heavy tables (every write updates all indexes)
  No Low cardinality columns (gender: M/F — only 2 values, index useless)
  No Small tables (full scan is fast anyway)
  No Columns rarely used in WHERE clauses
```

## Query Execution Plan (EXPLAIN)

Always verify your index is being used:

```sql
  EXPLAIN SELECT * FROM users WHERE email = 'a@gmail.com';

  Output:
  Index Scan using idx_users_email on users
  (cost=0.43..8.45 rows=1 width=80)
  → Yes Index is being used

  vs

  Seq Scan on users (cost=0.00..18540.00 rows=1000000)
  → No Full table scan — need an index!
```

## Index in Distributed Systems / Sharded Databases

When you shard a database, **each shard has its own local index**.

```
  Sharded by user_id:
  Shard 1: users 1–1M   → local index on email for shard 1
  Shard 2: users 1M–2M  → local index on email for shard 2

  Query: WHERE email = 'a@gmail.com'  ← which shard?
  → Must query ALL shards and merge results (expensive!)
  → Avoid queries that don't include the shard key
```

> Always design queries to **include the shard key** so only one shard is hit.

## Real-World Indexing Decisions

| System | Index On | Reason |
|---|---|---|
| **Twitter** | `tweet_id`, `user_id`, `created_at` | Timeline queries by user + time range |
| **Uber** | `driver_location` (geospatial) | Find nearby drivers |
| **Amazon** | `product_name` (full-text), `category_id` | Search and browse |
| **Banking** | `account_id`, `transaction_date` | Statement queries |
| **Netflix** | `user_id`, `content_id` | Viewing history lookup |

## Key Takeaways

- Index = data structure that speeds up **reads** at the cost of **slower writes and more storage**
- **B-Tree** is the default — good for equality and range queries
- **Hash** index is O(1) but **only for exact match**, no range queries
- **Composite index**: left-prefix rule — always filter on leftmost column
- **Full-text index** (inverted index) — powers search engines like Elasticsearch
- Don't over-index write-heavy tables — each index adds write overhead
- Use `EXPLAIN` to check if your index is actually being used
- In sharded systems, queries without the shard key must scatter to all shards
