# SQL vs NoSQL

---

## SQL (Relational)

Structured tables, fixed schema, powerful JOINs. Guarantees **ACID**.

| ACID Property | Meaning |
|---|---|
| Atomicity | All steps succeed or all fail (no partial commits) |
| Consistency | DB moves from one valid state to another |
| Isolation | Concurrent transactions don't interfere |
| Durability | Committed data survives crashes |

> Use SQL when data is relational, schema is stable, and transactions matter (payments, bookings).

---

## NoSQL — 4 Types

### 1. Key-Value Store
Simple `key → value` lookups. Extremely fast.
- **Redis, DynamoDB, Memcached**
- Use for: sessions, caching, rate-limit counters

### 2. Document Store
Data stored as JSON documents. No fixed schema. No JOINs — embed related data.
- **MongoDB, Firestore, CouchDB**
- Use for: product catalogs, user profiles, real-time apps

### 3. Wide-Column Store
Each row can have different columns. Optimised for massive write throughput.
- **Cassandra, HBase, ScyllaDB**
- Use for: IoT sensor data, time-series, event logs at scale

### 4. Graph Database
Nodes and edges — optimised for traversing relationships.
- **Neo4j, Amazon Neptune**
- Use for: social graphs, fraud detection, recommendations

---

## SQL vs NoSQL

| Dimension | SQL | NoSQL |
|---|---|---|
| Schema | Fixed | Flexible |
| Consistency | Strong (ACID) | Eventual (BASE) |
| Scaling | Vertical | Horizontal |
| Joins | Native | No joins (embed/denormalize) |
| Transactions | Full ACID | Limited |
| Best for | Relational, transactional | Massive scale, unstructured |

**BASE** (NoSQL trade-off): Basically Available, Soft state, Eventually consistent.

## Real-World Choices

| Company | SQL | NoSQL |
|---|---|---|
| Uber | PostgreSQL (trips) | Cassandra (GPS events) |
| Netflix | MySQL (billing) | Cassandra (viewing history) |
| Amazon | Aurora (orders) | DynamoDB (sessions, cart) |
| Instagram | PostgreSQL (users, posts) | Cassandra (follower graph) |

---

## Polyglot Persistence

Most large systems use multiple databases — the right tool per use case.

```
  E-Commerce:
  PostgreSQL   → Orders, Payments   (ACID required)
  MongoDB      → Product catalog    (flexible schema)
  Redis        → Sessions, cache    (fast key-value)
  Elasticsearch→ Product search     (full-text)
  Cassandra    → Clickstream logs   (write-heavy)
```

---

## Key Takeaways

- SQL = structured, relational, ACID — use for transactions
- NoSQL = flexible, horizontally scalable — use for scale and unstructured data
- Pick based on **access patterns and consistency requirements**, not preference
- In interviews: always justify the choice ("I'd use X because the access pattern is Y")
