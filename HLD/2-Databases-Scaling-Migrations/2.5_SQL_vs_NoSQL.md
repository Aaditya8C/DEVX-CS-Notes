# SQL vs NoSQL

---

## SQL (Relational Databases)

SQL databases store data in **structured tables** with rows and columns.

Data relationships are defined using **foreign keys** and queried using **JOIN** operations.

```
  Users Table:
  ┌────┬──────────┬───────────────────┐
  │ id │  name    │      email        │
  ├────┼──────────┼───────────────────┤
  │  1 │ Aaditya  │ a@gmail.com       │
  │  2 │ Rahul    │ r@gmail.com       │
  └────┴──────────┴───────────────────┘

  Orders Table:
  ┌────┬─────────┬────────┬────────┐
  │ id │ user_id │ amount │ status │
  ├────┼─────────┼────────┼────────┤
  │ 10 │    1    │  5000  │ placed │
  │ 11 │    1    │  2000  │  paid  │
  │ 12 │    2    │  8000  │  paid  │
  └────┴─────────┴────────┴────────┘

  Query: Get all orders for user "Aaditya"
  SELECT o.* FROM orders o JOIN users u ON o.user_id = u.id WHERE u.name = 'Aaditya';
```

---

## ACID Properties (SQL's Superpower)

SQL databases guarantee **ACID** properties for every transaction:

| Property | Meaning | Example |
|---|---|---|
| **Atomicity** | All operations in a transaction succeed or all fail | Transfer ₹1000: debit + credit both happen or neither |
| **Consistency** | DB moves from one valid state to another | Balance can never go negative (constraint enforced) |
| **Isolation** | Concurrent transactions don't interfere | Two users booking the last seat — only one succeeds |
| **Durability** | Committed data is never lost even on crash | Power cut after commit — data still there on restart |

> ACID is why **banking, payments, and financial systems always use SQL**.

## NoSQL Databases

NoSQL databases store data in **non-tabular formats** — designed for flexibility, scale, and speed.

"NoSQL" doesn't mean "no SQL" — it means **Not Only SQL**.

---

## Types of NoSQL Databases

### 1. Key-Value Store

Data stored as simple key → value pairs. Extremely fast lookups.

```
  "session:user123" → { userId: 123, cart: [...], expiry: ... }
  "product:42"      → { name: "iPhone", price: 79999 }
```

| Database | Use Case |
|---|---|
| **Redis** | Sessions, caching, rate limiting counters |
| **DynamoDB** | AWS-native, massive scale key-value |
| **Memcached** | Pure in-memory cache |

**When to use:** Caching, session storage, shopping carts, leaderboards.

---

### 2. Document Store

Data stored as **JSON/BSON documents**. Each document is self-contained. No fixed schema.

```json
{
  "_id": "order_123",
  "userId": "user_456",
  "items": [
    { "productId": "p1", "name": "Shoes", "qty": 2, "price": 3000 },
    { "productId": "p2", "name": "Shirt", "qty": 1, "price": 1500 }
  ],
  "address": {
    "street": "MG Road",
    "city": "Bangalore",
    "pin": "560001"
  },
  "total": 7500,
  "status": "shipped"
}
```

No need to JOIN tables — related data is **embedded** in one document.

| Database | Use Case |
|---|---|
| **MongoDB** | General purpose, product catalogs, user profiles |
| **CouchDB** | Offline-first mobile apps |
| **Firestore** | Real-time mobile/web apps (Firebase) |

**When to use:** Product catalogs, user profiles, content management, real-time apps.

---

### 3. Wide-Column Store (Column-Family)

Data stored in rows, but each row can have **different columns**. Optimized for massive writes and reads by key range.

```
  Row key: "user_123"
  Columns: { name: "Aaditya", age: 21, city: "Pune" }

  Row key: "user_456"
  Columns: { name: "Rahul", email: "r@g.com" }   ← different columns!
```

| Database | Use Case |
|---|---|
| **Cassandra** | Time-series data, IoT, write-heavy, global scale |
| **HBase** | Hadoop ecosystem, analytics |
| **ScyllaDB** | High-performance Cassandra replacement |

**When to use:** IoT sensor data, event logs, time-series metrics, write-heavy workloads at scale.

---

### 4. Graph Database

Data stored as **nodes and edges** — perfect for relationships and connections.

```
  (Aaditya) --[FOLLOWS]--> (Rahul)
  (Rahul)   --[FOLLOWS]--> (Priya)
  (Priya)   --[FOLLOWS]--> (Aaditya)

  Query: "Suggest friends to Aaditya" → Friends-of-friends in 2 hops
```

| Database | Use Case |
|---|---|
| **Neo4j** | Social networks, fraud detection |
| **Amazon Neptune** | Managed graph DB |

**When to use:** Social graphs, recommendation engines, fraud detection, knowledge graphs.

---

## BASE Properties (NoSQL Trade-off)

Most NoSQL databases follow **BASE** instead of ACID:

| Property | Meaning |
|---|---|
| **Basically Available** | System always responds (may be stale) |
| **Soft state** | State may change over time without input (replication lag) |
| **Eventually consistent** | Data will be consistent across nodes *eventually* |

```
  ACID (SQL):    Strong consistency, lower availability during failures
  BASE (NoSQL):  High availability, eventual consistency
```

---

## SQL vs NoSQL — Head-to-Head

| Dimension | SQL | NoSQL |
|---|---|---|
| **Schema** | Fixed (ALTER TABLE to change) | Flexible (each doc can differ) |
| **Consistency** | Strong (ACID) | Eventual (BASE) |
| **Scaling** | Vertical primarily (scale-up) | Horizontal (scale-out) |
| **Joins** | Native, powerful | No joins (embed or denormalize) |
| **Transactions** | Multi-table ACID | Limited (improving in Mongo, Cassandra) |
| **Query language** | SQL (standardized) | Varies per DB |
| **Best for** | Structured, relational data | Unstructured, flexible, massive scale |
| **Performance** | Predictable, good with indexes | Extremely fast for simple key lookups |

---

## When to Choose SQL

```
  Yes Data is highly relational (many JOINs needed)
  Yes ACID transactions are required (payments, banking, booking)
  Yes Schema is stable and well-defined
  Yes Complex queries and reporting (analytics, dashboards)
  Yes Compliance/auditability required
  Yes Team is strong in SQL

  Real-world: Banks, e-commerce orders, airline booking, inventory
```

---

## When to Choose NoSQL

```
  Yes Need horizontal scaling (billions of records)
  Yes Schema is flexible or evolving
  Yes Data doesn't have strong relationships
  Yes Write-heavy workload at scale
  Yes Need very low latency reads (caching, sessions)
  Yes Storing unstructured data (JSON docs, logs, events)
  Yes Global distribution (Cassandra, DynamoDB)

  Real-world: Social media feeds, product catalogs, IoT sensors, user sessions
```

## Real-World Database Choices

| Company | SQL Used | NoSQL Used | Why |
|---|---|---|---|
| **Uber** | PostgreSQL (trips) | Cassandra (location events) | Billions of GPS pings per day |
| **Instagram** | PostgreSQL (users, posts) | Cassandra (follower graph) | Social graph scale |
| **Netflix** | MySQL (billing) | Cassandra (viewing history) | Global write throughput |
| **Twitter** | MySQL (users) | Manhattan (custom KV) | Tweet reads at massive scale |
| **Amazon** | Aurora (orders) | DynamoDB (sessions, cart) | Sessions need extreme speed |
| **Airbnb** | MySQL (bookings) | Elasticsearch (search) | Full-text search on listings |

> Most large-scale systems use **both SQL and NoSQL** — the right tool for each job.

---

## Polyglot Persistence

Using multiple databases, each chosen for what it does best.

```
  E-Commerce System:

  PostgreSQL   → Orders, Payments (ACID required)
  MongoDB      → Product catalog (flexible schema)
  Redis        → Sessions, cart, caching (fast KV)
  Elasticsearch→ Product search (full-text)
  Cassandra    → Clickstream / activity logs (write-heavy)
```

---

## Key Takeaways

- **SQL** = structured, relational, ACID, scales vertically — use for transactions and relational data
- **NoSQL** = flexible, scalable horizontally, BASE — use for massive scale, unstructured data
- **4 types of NoSQL**: Key-Value, Document, Wide-Column, Graph — each solves a different problem
- **Don't pick randomly** — justify your choice based on the access patterns and consistency requirements
- Most production systems use **polyglot persistence** — multiple databases for different use cases
- In interviews: always state **"I'd use X because the access pattern is Y and we need Z property"**
