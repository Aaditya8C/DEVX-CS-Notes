# Publisher–Subscriber Model (Event-Driven Systems)

---

## Real-World Example: Ride Booking System (Uber/Ola-like)

### Components

- User Service (app)
- Ride Service
- Driver Service
- Notification Service
- Pricing Service
- Multiple backend servers
- Message Broker (Pub/Sub system)

---

## System Flow (High Level)

1. User requests a ride
2. Ride Service publishes an event: `RideRequested`
3. Multiple services subscribe and react independently:
   - Driver Service → finds nearby drivers
   - Pricing Service → calculates fare
   - Notification Service → notifies drivers
4. Driver accepts ride
5. Event published: `RideAccepted`
6. Other services react:
   - Notification Service → notify user
   - Tracking Service → start tracking trip

---

## Asynchronous Events

### Definition
Events are messages that represent **something that has happened** in the system.

Examples:
- RideRequested
- RideAccepted
- RideCompleted

---

### Characteristics

- Producers do not wait for consumers
- Events are processed independently
- Communication is non-blocking

---

## Publisher–Subscriber Model

### Definition
A messaging pattern where:

- **Publisher** emits events
- **Subscribers** listen to events and act on them

---

### Structure

Publisher → Message Broker → Subscribers

---

### In This Example

Publisher:
- Ride Service publishes `RideRequested`

Subscribers:
- Driver Service
- Pricing Service
- Notification Service

---

### Flow

1. Ride Service publishes event to broker
2. Broker distributes event to all subscribers
3. Each subscriber processes event independently

---

## Advantages of Event Systems

### 1. Loose Coupling
- Services do not depend on each other directly

---

### 2. Scalability
- Add more subscribers without changing publisher

Example:
- Add analytics service → just subscribe to events

---

### 3. Asynchronous Processing
- Improves system responsiveness

---

### 4. Fault Isolation
- Failure in one service does not break others

---

### 5. Extensibility
- Easy to introduce new features

Example:
- Add fraud detection by subscribing to events

---

## Drawbacks of Event Systems

### 1. Debugging Complexity
- Hard to trace event flow across services

---

### 2. Event Ordering Issues
- Events may arrive out of order

---

### 3. Duplicate Processing
- Same event may be processed multiple times

Solution:
- Idempotent consumers

---

### 4. Eventual Consistency
- Data may not be immediately consistent

---

### 5. Schema Evolution
- Changing event structure affects all subscribers

---

## Message Broker Role

- Receives events from publishers
- Distributes to subscribers
- Ensures durability and delivery

Examples:
- Kafka
- RabbitMQ
- AWS SNS/SQS

---

## Key Takeaways

- Pub/Sub enables asynchronous, event-driven communication
- Publishers and subscribers are decoupled
- Multiple services can react to the same event
- Improves scalability and flexibility
- Comes with trade-offs like complexity and eventual consistency
