# Database Sharding

## Definition
Database sharding is a technique used to **horizontally partition data across multiple databases or servers** so that each server stores only a subset of the total dataset.

Each partition is called a **shard**.

Sharding helps systems scale when a single database cannot handle increasing **data volume or query load**.

---

## Why Sharding is Needed

When a single database handles all data:

Clients → Application → Single Database

Problems:
- Limited storage capacity
- High query latency
- Increased contention for resources
- Difficult to scale horizontally

With sharding:

Clients → Application → Multiple Shards

Benefits:
- Data distributed across servers
- Reduced load on individual databases
- Improved performance and scalability

---

## Horizontal vs Vertical Partitioning

### Horizontal Partitioning (Sharding)

Rows are divided across multiple databases.

Example structure:

Shard 1 → User IDs 1–1,000,000  
Shard 2 → User IDs 1,000,001–2,000,000  
Shard 3 → User IDs 2,000,001–3,000,000

Characteristics:
- Same schema across shards
- Different subsets of rows

---

### Vertical Partitioning

Columns are split across different databases.

Example:

Database A → UserID, Name  
Database B → UserID, Address, Phone

Characteristics:
- Different columns stored separately
- Often used to separate frequently accessed data

---

## Shard Key

A **shard key** is the attribute used to determine which shard stores a specific row.

Examples:
- UserID
- CustomerID
- Region

Good shard key properties:
- High cardinality
- Even distribution
- Frequently used in queries

Poor shard key selection may lead to **data hotspots**.

---

## Sharding Strategies

### 1. Range-Based Sharding

Data is divided based on value ranges of the shard key.

Example:

Shard 1 → IDs 1–1000  
Shard 2 → IDs 1001–2000

Advantages:
- Simple implementation
- Efficient range queries

Disadvantages:
- Uneven data distribution possible

---

### 2. Hash-Based Sharding

Shard assignment determined using a hash function.

Shard = hash(key) % number_of_shards

Advantages:
- Even data distribution

Disadvantages:
- Difficult to perform range queries
- Adding new shards requires data redistribution

---

### 3. Directory-Based Sharding

A lookup service maintains a mapping of keys to shards.

Example:

UserID → Shard Mapping Table

Advantages:
- Flexible data placement
- Easy shard reassignment

Disadvantages:
- Requires maintaining an additional lookup service

---

## Rebalancing

When shards become uneven or new servers are added, data may need to be redistributed across shards.

Challenges:
- Data migration cost
- Temporary performance degradation
- Maintaining consistency during movement

Consistent hashing is often used to minimize redistribution.

---

## Challenges in Sharding

### Query Complexity
Queries involving multiple shards require **distributed queries**.

### Joins Across Shards
Joins become expensive or impractical.

### Data Rebalancing
Adding or removing shards requires data migration.

### Transaction Management
Maintaining ACID transactions across shards is difficult.

---

## Techniques to Reduce Sharding Complexity

- Use **denormalization**
- Avoid cross-shard joins
- Use **application-level routing**
- Use **consistent hashing for shard assignment**

---

## Real World Systems Using Sharding

Large-scale systems use sharding to scale databases:

Examples:
- Google Spanner
- Amazon DynamoDB
- MongoDB
- Cassandra

---

## Key Takeaways

- Sharding horizontally partitions data across multiple databases.
- Each shard stores a subset of rows with the same schema.
- Shard keys determine data placement.
- Common strategies include range-based, hash-based, and directory-based sharding.
- Sharding improves scalability but introduces operational complexity.
