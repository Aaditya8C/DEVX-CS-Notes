# CDN - Content Delivery Network

---

## The Problem: Global Access to a Single Server

Suppose a company's origin server is located in **US-East (Virginia)**.

```
                        [Origin Server]
                        US-East (Virginia)
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                    │
    Client (UK)         Client (India)       Client (Brazil)
    ~100ms RTT           ~200ms RTT           ~150ms RTT
```

- Every request travels all the way to the US server.
- Users in India, UK, Brazil suffer high latency.
- If the US server goes down, everyone is affected (SPOF).
- Heavy traffic from all regions hits a single point.

---

## Pre-CDN Solutions (and Their Limits)

### 1. Global Cache
- A single shared cache layer in front of the origin server.
- Reduces DB load, but the cache is still in one location.
- Users far from the cache still have high latency.

```
Client (India) → [Global Cache - US] → Origin Server
                  200ms RTT still applies
```

### 2. Shard Cache
- Cache is partitioned (sharded) by data type or key range.
- Reduces hotspot pressure per cache node.
- But all shards still live in one region — latency problem persists for remote users.

### 3. Localized Cache (Per-Region Cache)
- Deploy a cache server in each region (e.g., US, EU, Asia).
- Closer to users → lower latency.
- **Still requires the origin server for cache misses** — first request from each region hits the origin.
- Complex to keep caches consistent across regions.
- **Does not solve**: large static file delivery (images, JS, CSS, video), DDoS protection, or SSL termination at the edge.

### What Issue Persists?
- Static assets (images, videos, JS bundles) are large and fetched repeatedly.
- Even with localized caches, **every cache miss** goes back to the origin.
- High origin load during traffic spikes (e.g., viral video, product launch).
- No protection against DDoS at the network edge.

---

## What is a CDN

**Definition:** A geographically distributed network of **edge servers (PoPs — Points of Presence)** that cache and serve content from a location close to the user.

**What it solves:**
- Reduces latency by serving content from the nearest edge server.
- Offloads traffic from the origin server (static assets served from edge).
- Absorbs DDoS traffic at the edge before it reaches the origin.
- Handles SSL termination at the edge.
- Improves availability — even if origin is slow, cached content is served.

## CDN Architecture Diagram

```
                          [Origin Server]
                          US-East (Virginia)
                                 │
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
   [CDN Edge - London]  [CDN Edge - Mumbai]   [CDN Edge - São Paulo]
          │                      │                      │
   Client (UK)           Client (India)          Client (Brazil)
   ~10ms RTT              ~15ms RTT               ~12ms RTT
```

**Flow on first request (cache miss):**
```
Client (India) → CDN Edge (Mumbai) → Origin Server (US)
                                   ← Response cached at Mumbai edge
Client (India) ← CDN Edge (Mumbai) serves from cache
```

**Flow on subsequent requests (cache hit):**
```
Client (India) → CDN Edge (Mumbai) → serves directly from cache
                                     (Origin not touched)
```

## CDN Expiry Time (TTL - Time To Live)

- Each cached object has a **TTL** set via HTTP headers (`Cache-Control: max-age=<seconds>`).
- When TTL expires, the edge server fetches a fresh copy from the origin.

| Content Type | Typical TTL | Why |
|---|---|---|
| Static images / fonts | 1 day – 1 year | Rarely changes |
| JS / CSS bundles | 1 day – 1 year | Versioned via filename hash |
| HTML pages | 0 – 5 minutes | Changes frequently |
| API responses | 0 – 60 seconds | Dynamic data |
| Video chunks (HLS) | Minutes to hours | Large, stable segments |

**Cache Invalidation:**
- You can **purge** cached objects from CDN before TTL expires (e.g., after a deployment).
- Best practice: use **content-hashed filenames** (`app.abc123.js`) so new deploys auto-bust cache without manual purges.

## CDN Providers

| Provider | Notes |
|---|---|
| **Cloudflare** | Most popular; free tier, DDoS protection, DNS, Workers at edge |
| **AWS CloudFront** | Tightly integrated with AWS (S3, ALB, Lambda@Edge) |
| **Akamai** | Oldest, enterprise-grade, massive PoP network |
| **Fastly** | Real-time purging, popular with media companies |
| **Google Cloud CDN** | Integrated with GCP load balancers |
| **Azure CDN** | Integrated with Azure services |

---

## Summary

| Approach | Latency | Origin Load | Static Asset Delivery |
|---|---|---|---|
| Single origin server | High for remote users | Very high | Slow |
| Global cache | High (still one region) | Reduced | Slow |
| Shard cache | High (still one region) | Reduced | Slow |
| Localized cache | Medium (misses hit origin) | Moderate | Moderate |
| CDN | Low (edge is nearby) | Very low | Fast |
