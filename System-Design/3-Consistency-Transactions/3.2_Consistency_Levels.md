# Consistency Levels in Distributed Systems

## Definition
Consistency levels define **how up-to-date and synchronized data must be across replicas when performing read/write operations**.

They determine the trade-off between:
- Consistency
- Availability
- Latency

---

# Types of Consistency Levels

## 1. Strong Consistency (Linearizability)

### Definition
After a write completes, **all subsequent reads return the latest value**.

### Guarantee
- Single global order of operations
- No stale reads

### Example
User updates profile name → immediately reads → always sees updated name

### Use Cases
- Banking systems
- Payment systems
- Inventory management

### Drawbacks
- High latency
- Lower availability (during failures)

---

## 2. Sequential Consistency

### Definition
Operations are executed in the **same order across all nodes**, but not necessarily in real-time order.

### Guarantee
- Same order for all clients
- May not reflect real-time order

### Example
Two writes:
- W1 → W2

All nodes see W1 before W2, but timing may vary.

### Use Case
- Collaborative systems

---

## 3. Causal Consistency

### Definition
Operations that are **causally related must be seen in order**, but unrelated operations may be seen differently.

### Guarantee
- Preserves cause-effect relationship

### Example
User A posts a tweet → User B replies

All users must see:
- Tweet before reply

But independent tweets may appear in different orders.

### Use Cases
- Social media
- Messaging systems

---

## 4. Eventual Consistency

### Definition
If no new updates occur, all replicas will **eventually converge to the same value**.

### Guarantee
- No strict ordering
- Temporary inconsistency allowed

### Example
User updates status:
- Some users see old status briefly
- Eventually all see updated status

### Use Cases
- DNS
- Caching systems
- Social feeds

---

## 5. Read-Your-Writes Consistency

### Definition
A user always sees their **own latest writes**.

### Example
User updates profile → refreshes → sees updated value

Even if others see stale data.

### Use Case
- User dashboards
- Personal settings

---

## 6. Monotonic Reads

### Definition
Once a user sees a value, they will **never see an older value later**.

### Example
User reads version 5 → later reads → cannot see version 3

---

## 7. Monotonic Writes

### Definition
Writes from a single user are applied in **order**.

### Example
Write1 → Write2

System ensures Write1 is applied before Write2

---

## 8. Writes Follow Reads

### Definition
A write operation is based on the **latest read value**.

### Example
User reads version 5 → updates → system ensures update is based on version 5

---

# Quorum-Based Consistency

## Definition
Uses majority voting for reads and writes.

Let:
- N = total replicas
- W = write quorum
- R = read quorum

Condition for strong consistency:

R + W > N

---

## Example

N = 3 replicas

- W = 2 (write to 2 replicas)
- R = 2 (read from 2 replicas)

Guarantee:
- At least one overlapping node ensures latest data

---

## Trade-offs

| Configuration | Result |
|--------------|--------|
| W = N, R = 1 | Strong writes, fast reads |
| W = 1, R = N | Fast writes, strong reads |
| W + R ≤ N | Weak consistency |

---

# Comparison

| Consistency Level | Freshness | Latency | Availability |
|------------------|----------|--------|--------------|
| Strong | Always latest | High | Low |
| Sequential | Ordered | Medium | Medium |
| Causal | Partial order | Medium | High |
| Eventual | May be stale | Low | High |

---

# Key Trade-offs

- Strong consistency → accuracy over availability
- Eventual consistency → availability over accuracy
- Causal consistency → balance for user experience

---

# Real World Mapping

- Banking → Strong consistency
- WhatsApp → Causal consistency
- Instagram feed → Eventual consistency
- DNS → Eventual consistency

---

# Key Takeaways

- Consistency levels define how up-to-date data is across nodes
- Strong consistency ensures correctness but increases latency
- Eventual consistency improves availability but allows stale reads
- Quorum-based systems balance consistency and performance
- Choice depends on application requirements
