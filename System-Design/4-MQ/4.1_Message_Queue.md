# Asynchronous Systems & Message Queues (Pizza Shop Example)

---

## Pizza Shop Example (Synchronous vs Asynchronous)

### Synchronous Flow (Blocking)

Customer → Order → Pizza made → Delivered → Next order

Problems:
- Customer must wait until entire process completes
- Shop handles one order at a time
- Poor scalability

---

### Asynchronous Flow (Using Queue)

Customer → Order placed → Added to queue → Worker prepares pizza → Delivered

- Order is accepted immediately
- Processing happens in background

---

## Benefits of Asynchronous Processing

### 1. Improved Responsiveness
- System responds immediately without waiting for processing

Example:
- Order confirmation shown instantly

---

### 2. Better Throughput
- Multiple tasks processed in parallel

---

### 3. Loose Coupling
- Producer (order service) and consumer (kitchen) are independent

---

### 4. Fault Isolation
- Failure in one component does not crash entire system

---

### 5. Retry Mechanism
- Failed tasks can be retried later

---
## Scaling Shops (Horizontal Scaling)

### Definition
Scaling shops means **adding more pizza shops (workers/services)** to handle increasing demand.

---

### Example

- 1 shop → handles 50 orders/hour  
- 5 shops → handle 250 orders/hour  

System:

Orders → Queue → Shop1  
                 → Shop2  
                 → Shop3  
                 → Shop4  
                 → Shop5  

---

### How It Works

- All shops consume from the same queue
- Each shop picks up orders independently
- Load is automatically distributed

---

### Benefits

- Handles traffic spikes
- No single bottleneck
- Easy to scale up/down dynamically

---

## Fault Tolerance (Shop Failure Handling)

### Scenario

- Shop2 crashes while processing an order

---

### What Happens?

1. Shop2 does NOT acknowledge (ACK) the message
2. Message remains in queue (or becomes visible again)
3. Another shop (e.g., Shop3) picks up the same order

---

### Key Mechanisms

#### 1. Acknowledgement (ACK/NACK)

- ACK → message processed successfully → removed from queue  
- No ACK → message is retried

---

#### 2. Visibility Timeout

- When a shop picks a message:
  - It becomes temporarily invisible to others
- If not ACKed within timeout:
  - Message becomes visible again

---

#### 3. Retry Mechanism

- Failed orders are retried automatically

---

#### 4. Dead Letter Queue (DLQ)

- If message fails repeatedly:
  - Moved to DLQ for manual inspection

---

### Result

- No order is lost
- Failed shop does not impact entire system
- System self-recovers

---

## Features of a Message Queue

### 1. Durability
- Messages are stored safely (not lost on failure)

---

### 2. Ordering (Optional)
- FIFO (first-in-first-out) in some systems

---

### 3. Delivery Guarantees

- At-most-once → may lose messages
- At-least-once → may duplicate
- Exactly-once → complex but ideal

---

### 4. Scalability
- Multiple producers and consumers

---

### 5. Decoupling
- Services do not directly depend on each other

---

### 6. Retry & DLQ
- Failed messages are retried or moved to dead-letter queue

---

## Encapsulation

### Definition
Each component handles its own responsibility without exposing internal logic

---

### In Pizza Example

- Order Service → only places order
- Queue → handles delivery of tasks
- Worker → processes order

---

### Benefit

- Easier maintenance
- Independent scaling
- Reduced system complexity

---

## Final Flow (Putting It All Together)

1. Customer places order
2. Order service pushes message to queue
3. Queue stores order reliably
4. Workers consume and process orders
5. Failures handled via retries

---

## Key Takeaways

- Asynchronous systems improve scalability and responsiveness
- Message queues enable decoupling and fault tolerance
- Horizontal scaling allows handling large workloads
- Encapsulation keeps systems modular and maintainable
